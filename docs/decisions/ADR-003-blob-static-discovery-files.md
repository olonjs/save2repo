# ADR-003: Generate and serve tenant discovery files from Vercel Blob on hot save

## Status

Proposed

## Date

2026-05-13

## Context

Tenant sites expose a set of discovery and machine-readable files at well-known paths:

| File | Path | Purpose |
|---|---|---|
| `robots.txt` | `/robots.txt` | Crawler directives |
| `sitemap.xml` | `/sitemap.xml` | Page index for crawlers |
| `llms.txt` | `/llms.txt` | LLM agent entry point |
| `mcp-manifest.json` | `/mcp-manifest.json` | Site-level MCP manifest index |
| Page JSON | `/$slug.json` → `/pages/$slug.json` | Raw page content for agents |
| Page manifests | `/mcp-manifests/$slug.json` | Per-page MCP manifest |

These files are currently generated by scripts (`robots.mjs`, `sitemap.mjs`, `generate-llms-txt.mjs`, `bake.mjs`) that run as part of the Vite production build and are published as static assets on Vercel.

**The drift problem:** when a tenant operator performs a hot save (content is written to `tenant_content_store` in Supabase), the deployed static files are not updated. They reflect the state of the last cold save + rebuild. The gap between hot save and the next cold save can be hours or indefinite. During this window:

- `llms.txt` and `mcp-manifest.json` list stale page titles, descriptions, and section types — AI agents reading the site receive outdated structure.
- `sitemap.xml` may reference non-existent or renamed pages, or miss new ones.
- `pages/$slug.json` returns the pre-save content to any agent fetching it directly.

The platform already has the full content in `tenant_content_store` immediately after hot save. The generation logic in `@olonjs/core` (`buildLlmsTxt`, `buildSiteManifest`, `buildPageManifest`) is pure TypeScript with no React or Vite dependencies and only requires `zod` (already a platform dependency).

Files that are deliberately **excluded** from this mechanism:

- `schemas/$slug.schema.json` — these depend on Zod schema instances defined in the tenant's DNA (section components), which are unavailable on the platform. They drift with DNA changes, not content changes, so cold save + rebuild is the correct update path.

## Decision

On completion of every successful hot save (`save2edge-snapshot`), the platform generates the six file families listed above in memory and uploads them to the Vercel Blob store at stable, deterministic paths under `tenants/{tenantSlug}/`. The tenant template gains an Edge Middleware that intercepts requests for these paths and serves from Blob.

**Generation runs synchronously as a final step in the existing SSE flow** (`save2edge-snapshot`) — visible to the operator as a `static_files` step. If generation or upload fails, the error is logged and emitted as a non-fatal SSE `log` event; the hot save is not rolled back.

**Blob path scheme** (all with `addRandomSuffix: false` for stable URLs):

```
tenants/{slug}/robots.txt
tenants/{slug}/sitemap.xml
tenants/{slug}/llms.txt
tenants/{slug}/mcp-manifest.json
tenants/{slug}/pages/{page-slug}.json          (one per page)
tenants/{slug}/mcp-manifests/{page-slug}.json  (one per page)
```

**Base URL:** `robots.txt` and `sitemap.xml` require the tenant's public URL. Resolution order: (1) active/verified custom domain from `tenant_domains` — requires an additional query at generation time; (2) `tenants.vercel_public_url`; (3) `tenants.vercel_url`. The first non-null value is used.

**Builder code:** `buildLlmsTxt`, `buildSiteManifest`, `buildPageManifest`, `buildPageContract` and supporting helpers are copied from `@olonjs/core/src/contract/webmcp-contracts.ts` into `src/lib/webmcpBuilders.ts` on the platform. `schemas` is passed as `{}` — section schemas are not available at the platform level and are not needed for content-derived files.

**Tenant serving:** the tenant DNA template (`tenant-radice`) contains a `vercel.json` with all necessary rewrites already defined, using two placeholders: `{BLOB_BASE}` and `{slug}`. During the `repo` step of `provision-stream` (after the template files are copied to the new GitHub repo, **before** the first Vercel deploy is triggered), the platform reads the `vercel.json` from the newly created repo, replaces `{BLOB_BASE}` with `JSONPAGES_BLOB_PUBLIC_BASE` and `{slug}` with `vercelSlug`, and commits the resolved file back to the repo. One deploy is triggered, already with the correct rewrites. The `vercel.json` is never modified again after provisioning; Blob content changes (hot saves) do not require a new deploy because the destination URLs are stable (`addRandomSuffix: false`).

## Alternatives Considered

### Do nothing (status quo)

- Pros: no new infrastructure or template changes.
- Cons: drift is permanent until the operator explicitly runs a cold save. LLM crawlers and MCP agents reading the site after a Studio edit receive stale data.
- Rejected: the drift is a functional correctness issue for the agentic surface, not just a cosmetic one.

### Regenerate on cold save only

- Pros: no change to the hot save flow.
- Cons: cold save already triggers a full Vite rebuild which regenerates these files. This option would add redundant Blob uploads with no user benefit. Does not solve the hot-save drift.
- Rejected: same as do nothing for the hot save window.

### Serve dynamically through the platform (`/api/v1/tenants/:id/static/:file`)

- Pros: no Blob, no tenant middleware; the platform is the sole serving layer.
- Cons: every crawler hit on `robots.txt`, `sitemap.xml`, `llms.txt` adds a round-trip through the platform. These endpoints are hit by Googlebot, GPTBot, ClaudeBot etc. — potentially high frequency. Adds load and latency to the platform for files that can be static.
- Rejected: CDN-served Blob is more appropriate for static-content delivery.

### Async queue / background job

- Pros: decouples generation from the hot save latency.
- Cons: introduces a residual drift window (seconds to minutes) plus more infrastructure (queue, worker). The generation takes < 100 ms in memory; there is no latency argument for deferring it.
- Rejected: synchronous in-process generation is simpler and eliminates the drift window entirely.

### Edge Middleware fetches from platform on demand (no Blob)

- Pros: always 100% fresh; no generation step.
- Cons: same concerns as serving dynamically through the platform. Every crawler request cold-fetches from the platform.
- Rejected: Blob is the correct layer for CDN-cached static content.

### Install `@olonjs/core` on the platform

- Pros: single source of truth for builder functions.
- Cons: `@olonjs/core` bundles React, Vite plugins, and SSR runtime code. Installing it on the platform risks build size bloat, peer-dependency conflicts, and pulling in browser-only modules into the Next.js server runtime.
- Rejected: copying the 350-line `webmcp-contracts.ts` file (pure TS, only `zod`) is simpler and avoids the dependency surface. The file must be kept in sync manually when upstream changes; a comment in the copy references the source.

## Consequences

**Easier:**
- Hot save makes the agentic surface (`llms.txt`, MCP manifests, page JSON) immediately consistent with the content DB — no operator action required.
- `sitemap.xml` and `robots.txt` stay current without a rebuild.

**Harder:**
- `webmcp-contracts.ts` is now duplicated in the platform (`src/lib/webmcpBuilders.ts`). Changes to builder logic in `@olonjs/core` must be mirrored manually. A comment in the copy and a note in the tenant repo's CLAUDE.md make this explicit.
- The tenant template gains an Edge Middleware dependency on the Blob store. If the Blob store is unavailable, these six file families return errors until it recovers. The previous static files (from the last build) are no longer served as fallback.
- `VITE_BLOB_TENANT_BASE_URL` must be injected at provision time; existing tenants need a one-time backfill of the env var and an initial Blob upload.

## Follow-ups

- Backfill script for existing tenants: trigger a Blob upload from their current `tenant_content_store` and inject `VITE_BLOB_TENANT_BASE_URL`.
- Cold save should skip re-uploading to Blob for these files (build already regenerates them and writes to `public/`) — or accept the redundant upload as a correctness guarantee.
- Evaluate whether `schemas/$slug.schema.json` can be served from the Blob artifact written during cold save, making the schema path also drift-free after builds.
